# 总盘研判流程

如果需要共享术语、读盘抓手或冲突裁定口径，读取 `术语框架.md`。

## 适用场景

当用户要的是整张盘的系统拆盘，而不是单一专题时，使用这个流程。

典型请求：

- 完整命盘研判
- 本命总览
- 命主盘（Rashi chart）与九分盘（Navamsha chart）联读
- 既往事件回看
- 出生时间稳定度判断
- 基于 Jagannatha Hora PDF 或截图做基础命盘拆盘

如果用户只想看事业、婚姻，或地点与时间窗口的专项问题，不要走完整总盘，改读 `事业.md`、`婚姻.md` 或 `窗口与场域.md`。

## 需要的输入

接受以下任一资料：

- Jagannatha Hora 导出的 PDF
- JHora 导出的 markdown 报告
- 星盘截图
- 文本形式的行星位置
- 当前对话里已经存在的总盘摘要

能提取到的话，优先确认这些字段：

- 上升星座与度数
- 九大行星的星座、宫位、度数、逆行状态
- Nakshatra 和 pada
- 九分盘（Navamsha chart）落点与上升
- Vimshottari dasha 时间线
- Shadbala、SAV、BAV
- Chara karaka

不要因为某张可选表缺失就卡死整份解读。只在关键字段缺失时追问。

## 计算纪律

只要原始资料里有数字，就把可确定的检查交给本地 Python。

必须工具化的项目：

- SAV、BAV 算术检查
- 根据度数反推 Nakshatra 和 pada
- 根据度数反推九分盘（Navamsha chart）落座
- Mercury、Venus 与 Sun 的距角
- Rahu、Ketu 对冲检查
- 逆行合法性
- dasha 年月区间运算

优先使用：

```bash
python "scripts/chart_sanity_check.py" <chart-input.json-or-jhora.md>
```

如果是 JHora 导出的 markdown，优先直接把原文件送进脚本，不要先手工转述一遍。
如果 SAV 或 BAV 所在的图式表格无法稳定结构化，就先保住基础检查，把矩阵类检查标成跳过或待补整理。

如果只有部分数字，就只检查已有部分，并在结论里明确哪些项目被跳过。

## 工作流

### 1. 先整理基础盘面

在解释前先建立一个干净的底稿：

- 资料来源和完整度
- 上升
- 行星落点
- 当前所处的大运阶段
- 九分盘（Navamsha chart）数据
- 可用的强弱表

如果 OCR 或截图识别有歧义，要明确指出哪一个值不确定。

### 2. 做一致性检查

基础检查始终执行：

- 九大行星是否齐全且不重复
- Rahu 与 Ketu 是否形成对冲轴
- Mercury 与 Venus 是否和 Sun 保持合理距角
- 在数据足够时，Nakshatra 与 pada 是否和度数一致
- 在数据足够时，九分盘（Navamsha chart）映射是否和主盘度数一致
- dasha 序列是否内部自洽

SAV/BAV 检查是**必须尝试、失败才跳过**的项目，不是可选的"加分项"：

- 只要原始资料包含任何 SAV/BAV 数字（PDF 截图、JHora 导出的表格、用户手动输入），就必须提取并完成三项校验
- 如果在 PDF 或 markdown 中能肉眼辨认出数字，优先手动列出 12 宫 SAV 值，再做算术校验
- 只有经过尝试后表格确实无法结构化（如 OCR 输出乱码、截图模糊不可辨认），才标记为"跳过"
- 跳过时必须在回执中说明具体原因（"截图分辨率不足"而非笼统的"图式表格未能自动化解析"）

**强矩阵检查**（SAV/BAV 数据可用时）：

- 12 宫 SAV 总和是否为 337（详见 `术语框架.md` SAV/BAV 量化规范）
- 每颗星的 BAV 行总和是否符合固定常数
- 每个星座的 BAV 列和是否等于对应 SAV
- 行列同时出错时，用交叉定位找可能的 OCR 或录入错误

**降级版**（SAV/BAV 经尝试确实无法提取时）：
- 保留所有基础检查
- 明确写 "SAV/BAV ⏭️ 跳过（原因：...），宫位级量化判断降级为 🟠 中"
- 后续的"资源水位"和"落点环境"镜面使用 Shadbala 和 Vimsopaka 作为替代数据源

每次一致性检查后，**必须输出以下编号校验表**。这是硬性输出格式，缺任何一行都视为校验不完整：

```
=== 一致性检查（必须逐项给出结果，不可省略任何一行）===

C1  行星完整性   : ✅/❌  [说明]
C2  Nakshatra映射: ✅/❌  [说明]
C3  D9映射       : ✅/❌  [说明]
C4  日水距角     : ✅/❌  [度数+判断]
C5  日金距角     : ✅/❌  [度数+判断]
C6  罗计对冲     : ✅/❌  [说明]
C7  逆行合法性   : ✅/❌  [说明]
C8  Dasha序列    : ✅/❌  [说明]
C9  SAV总和=337  : ✅/❌/⏭️  [数值+说明；如跳过必须写明具体原因]
C10 BAV行常量   : ✅/❌/⏭️  [7/7几颗匹配；如跳过必须写明具体原因]
C11 BAV列→SAV   : ✅/❌/⏭️  [12/12几宫匹配；如跳过必须写明具体原因]

校验等级：🔴极高(11/11通过) / 🟡高(C1-C8全过,C9-C11跳过) / 🟠中(有硬冲突)
```

**C9-C11 的硬规则**：
- 只要命盘资料中包含任何 SAV/BAV 数字（无论格式），就必须逐项给出结果
- 脚本解析不了 → 手动从 PDF/截图/HTML 中读取 12 个 SAV 值，做加法验证
- 确实无法提取（OCR 乱码、截图模糊不可辨）→ 标记为 ⏭️ 并写明具体原因
- 禁止出现"SAV/BAV 跳过"而无任何数字输出——要么给出具体数值和校验结果，要么给出具体跳过原因

校验等级影响后续分析的置信度标注：
- 🔴极高：所有宫位级判断正常使用 SAV 阈值
- 🟡高：宫位级判断改用 Bhava Bala + Shadbala 替代，"资源水位"和"落点环境"镜面降级

### C1-C11 计算公式（AI 自算用）

以下公式用于 AI 从原始度数数据自行计算校验值，而非仅检查 JHora 已有标注。**OCR 场景下标注可能读错但度数是准的——AI 必须会自己算。**

**C2 Nakshatra 与 Pada 计算**：
```
绝对经度 = 星座序号 × 30 + 度数    （白羊=0, 金牛=1, ..., 双鱼=11）
Nakshatra编号 = floor(绝对经度 / 13.3333) + 1
Pada编号 = floor((绝对经度 % 13.3333) / 3.3333) + 1
```
计算结果与 JHora 标注对比。不匹配 → ❌，列出不一致的星体。

**C3 D9 映射计算**：
```
 Pada → D9 星座映射：
   火象星座(白羊1/狮子5/射手9) pada起点 → 白羊
   土象星座(金牛2/处女6/摩羯10) pada起点 → 摩羯
   风象星座(双子3/天秤7/水瓶11) pada起点 → 天秤
   水象星座(巨蟹4/天蝎8/双鱼12) pada起点 → 巨蟹
 Pada编号n对应的D9星座 = 起点星座序号 + (n-1)
```
与 JHora 的 D9 盘标注对比。不匹配 → ❌，D1 度数或 D9 盘至少一个读错。

**C4/C5 水金距角计算**：
```
|Mercury绝对经度 - Sun绝对经度| ≤ 28°（最大距角）
|Venus绝对经度 - Sun绝对经度| ≤ 48°（最大距角）
```
超出容差 → ❌，行星符号可能 OCR 混淆。

**C6 罗计对冲验证**：
```
|Rahu绝对经度 - Ketu绝对经度| = 180°（容差 ±1°）
且落宫差 = 6
```
不满足 → ❌，节点位置读取错误。

**C7 逆行合法性**：
```
Sun/Moon 永远不逆行 → 出现逆行标记为数据错误
Rahu/Ketu 永远逆行 → 未标记逆行需二次确认
```

**C8 Dasha 间隔常数链**：
```
相邻 Mahadasha 年份差：
  Sun(6年) → Moon(10年) → Mars(7年) → Rahu(18年) →
  Jupiter(16年) → Saturn(19年) → Mercury(17年) → Ketu(7年) → Venus(20年)
验证：Date(n+1) - Date(n) = 对应常数年
```

**C9 SAV 总和**：
```
Σ(SAV₁ + SAV₂ + ... + SAV₁₂) = 337
```
不通过 → 逐宫核实数值，定位读错宫位。

**C10 BAV 行常量**：
```
Sun=48 | Moon=49 | Mars=39 | Mercury=54 | Jupiter=56 | Venus=52 | Saturn=39
对每颗星求 BAV 12 宫之和，匹配常数。
```

**C11 BAV 列→SAV 交叉验证**：
```
∀宫位i: BAV(Sun,i) + BAV(Moon,i) + ... + BAV(Saturn,i) = SAV(i)
```
不通过 → C10 和 C11 同时出错的格子即为错误点，交叉定位。

### 3. 做既往事件回看

既往事件回看不是附带动作，而是一个主动的可信度检查。其目的不仅是判断时间结构是否可信，更是校准推断系统——诊断「如果回顾的推断与事实不同，断在了推断链的哪一环」。

先判断当前是哪一种模式：

- `提问模式`：用户还没给出既往事件，只能由你先提 2 到 4 个高信号窗口
- `直评模式`：用户已经给出明确时间线，需要你逐条回看，不要再改写成连环追问

**超出旧命中等级的诊断维度**（见 SKILL.md 分层推断规则层 3）：

既往事件回看不再仅使用「高贴合/有限贴合/失配」三档对错标签。每条事件必须额外输出三个诊断维度：

| 维度 | 评估 |
|------|------|
| **信号命中** | 星盘在该时间段是否确实有对应类型的结构信号激活？✅ 命中 / ⚠️ 部分命中 / ❌ 未命中 |
| **推断合理性** | 给定当时的星盘信号 + 背景事实，从信号到具体推断的逻辑链是否成立？✅ 合理 / ⚠️ 方向对但精度偏 / ❌ 逻辑有缺陷 |
| **信息完备度** | 推断时所掌握的层 1 背景事实是否足够支撑这个精度的结论？🔴 高 / 🟡 中 / ⚪ 低 |

提问模式推荐顺序：

1. 先判断用户提供的出生时间精度
2. 收集该领域背景事实卡（按需，只收集当前分析方向所需的信息）
3. 在过去 5 到 10 年里选 2 到 4 个高信号窗口
4. 对每个窗口，结合背景事实卡做分层推断（层 0 信号 → 层 1 背景 → 层 2 具体假设）
5. 让用户逐条确认、否认或部分确认
6. 用诊断维度复盘：哪些信号确实命中了？推断链中哪一环需要修正？

设问规则：

- 一定落到真实年份或年份区间
- 一定问具体事件类型
- 只给足够的依据，不要写成长篇解释
- 设问本身已经是一轮小型分层推断——先展示层 0 信号，再说明结合了什么背景信息得出的假设

直评模式的最低输出格式：

```text
事件：[用户给出的具体事件]
├── 命中等级（旧标签兼容）：高贴合 / 有限贴合 / 失配
├── 信号命中：✅ / ⚠️ / ❌
├── 推断合理性：✅ / ⚠️ / ❌
├── 信息完备度：🔴 / 🟡 / ⚪
├── 时间锚点：哪段 dasha 或哪组时间结构在支撑
├── 盘面锚点：至少两类证据
├── 牵强处：这条结论哪里不能写得太满
└── 诊断备注：[如果命中等级不是高贴合，说明三个维度中哪个出了问题]
```

如果用户给了 5 条重大事件，就至少逐条覆盖 5 条，不要只挑最容易命中的两三条。

**诊断汇总**：逐条回看完成后，必须输出一个简短的诊断汇总，说明既往事件中信号命中率、推断链的主要断点类型、以及信息完备度对精度的影响。

### 4. 判断出生时间稳定度

综合看三件事：

- 用户自己说的时间精度
- 上升或宫位边界在附近时间窗内会不会变化
- 既往事件回看和盘面逻辑的匹配程度

如果出生时间不稳：

- 降低宫位级判断的置信度
- 保留行星落星座与较粗粒度的人格结论
- 明说哪些部分稳定，哪些部分可能漂移

出生时间稳定度判断，不能只凭”感觉像”。要结合：

- 一致性检查有没有硬冲突
- 既往事件回看的信号命中率（✅ 命中占比）与推断合理性分布
- 诊断维度异常的聚集模式：信号未命中集中在哪类事件、推断偏差集中在哪个领域（结构/窗口/专名）

### 验证闸门：通过才能进入完整拆盘

步骤 1-4（基础盘面 → 一致性检查 → 既往事件回看 → 出生时间稳定度）构成一个验证闭环。**在这个闭环闭合之前，不允许进入步骤 5-8 的完整拆盘交付，也不允许生成 `report.html`。**

闸门禁止的是 **交付级完整拆盘**，不禁止步骤 3 本身需要的盘面扫描。区分如下：

- **允许**（服务于步骤 3 提问质量）：扫描 dasha 时间线找切换点、定位过去 5-10 年激活的宫位和 karaka、识别高信号窗口。这些是生成靠谱提问的前提，不做反而会导致问题空泛无效。
- **禁止**（交付物）：展开 9 颗行星的八镜完整拆盘（步骤 5）、九分盘兑现复核（步骤 6）、人生主线汇总（步骤 7）、生成 `report.html`。

一句话：**可以用盘面数据来提问，但在用户确认信号命中之前不要写成报告。**

验证闭环的最小通过条件：

- 一致性检查没有硬冲突（行星数量不对、罗计轴断裂、距角异常等）
- 既往事件回看中，信号命中 ≥ 1 条 `✅ 命中` 或 `⚠️ 部分命中`（信号不能全部未命中——那说明时间结构本身有问题）
- 且推断合理性 `✅ 合理` 或 `⚠️ 方向对但精度偏` 的比例 ≥ 50%（推断逻辑至少有一半以上站得住）
- 出生时间稳定度不低于 `中等`

**注意**：闸门核心条件已从旧版本的「至少一条高贴合」改为上述复合条件。「高贴合」是结果层面的对错标签，而信号命中 + 推断合理性才是系统可信度的真正指标。信号命中但精度偏，说明系统本身没问题、背景信息不够——这不应导致闸门关闭。

如果验证闸门未通过：

- **不要继续展开 9 颗行星完整拆盘**（步骤 5-6）
- **不要汇总人生主线**（步骤 7）
- **不要生成 `report.html`**
- 降级为交付物：基础盘面摘要 + 一致性检查回执 + 既往事件回看结果（含诊断汇总） + 出生时间稳定度判断 + 初步方向提示
- 明确告知用户闸门卡在哪一项：是信号全未命中（时间问题），还是推断合理性大面积偏/失配（配置解读问题），还是出生时间稳定度不够

验证闸门通过后，才依次推进步骤 5-8，并将步骤 1-4 的结果作为置信度基准写入最终报告。

**闸门确认提示（必须在聊天中输出，不可跳过）**：

```
=== 验证闭环完成 ===
闸门状态：[通过 / 未通过]
- 一致性检查：C1-C11 通过 X/11 项
- 既往事件回看：
  信号命中率：✅ X 条 / ⚠️ X 条 / ❌ X 条
  推断合理性：✅ X 条 / ⚠️ X 条 / ❌ X 条
  信息完备度：🔴 X 条 / 🟡 X 条 / ⚪ X 条
  （旧标签兼容：高贴合 X 条 / 有限贴合 X 条 / 失配 X 条）
- 诊断汇总：[主要断点类型——信号？推断？信息？]
- 出生时间稳定度：[稳定 / 中等 / 明显不稳]
- 校验等级：[🔴极高 / 🟡高 / 🟠中]

请确认以上验证结果。
回复"继续"→ 进入完整拆盘（第二阶段）
回复具体疑问 → 针对性补充
===
```

### 5. 展开命主盘（Rashi chart）行星拆盘

对于完整拆盘，命主盘行星部分是主要交付物，不是隐藏在后面的辅助说明。

9 颗行星都要覆盖，但分组和篇幅深度不由固定分类决定——由 SKILL.md「信号分级与篇幅控制」动态定级。输出顺序按 A→B→C 信号级别组织（同级内按 Shadbala 降序）。

八镜项目框架（A 级全写，B/C 级按信号分级规则缩减；"运行状态"在 A/B 级必须展开子维度）：

- 主题归属
- 运行状态（**必须包含**：是否燃烧及距日度数、是否逆行及修正效果；如有 JHora 数据则追加年龄状态和情绪/活动状态）
- 资源水位（优先使用掌宫 SAV；如 SAV 不可用，使用 Shadbala 百分比 + Vimsopaka）
- 落点环境（优先使用落宫 SAV 分档；如 SAV 不可用，使用 Bhava Bala 或宫位性质定性评估）
- 兑现通道
- 外力牵引
- 熟成节律
- 格局放大（如有 yoga 参与）

**附加硬性检查**（不写入镜面表格则视为遗漏）：
- 如果行星掌管角宫（1/4/7/10），必须检查角宫主效应（Kendradhipati Dosha），见 `术语框架.md`
- 如果行星参与互溶（Parivartana），必须在"格局放大"镜面中展开互溶双方的关系
- 如果行星是某段 dasha 的主星，必须在"熟成节律"中标注对应的 dasha 时段

建议按这个格式输出：

```text
——— [行星名] ———
【现实判断】
2 到 3 段短文，交代它在你人生里到底管什么、怎么发力、当前状态如何。

【盘面抓手】
```

### 5.5 互溶与格局扫描（必做）

在完成 9 颗行星的独立拆盘后，必须进行一次"跨行星扫描"，检查行星间的联动关系。这不是可选的"加分分析"，而是结构判断的核心输入。

**互溶（Parivartana）检查**：

全盘逐对检查：是否有两颗行星互相落在对方的星座？

- 如果有互溶，这是全盘最重要的格局之一，必须单独展开分析
- 最低要交代：互溶双方各自的宫位掌管（互换了什么主题）、双方各自的落宫状态、互溶的"交易"是即时兑现还是延迟兑现
- 如果互溶双方的 dasha 时段有先后 → 标注兑现时间线

**互溶动力学（强弱判定）**：

互溶不是简单的"1+1=2"。两颗星通过守护关系建立了能量通道，通道的效率取决于双方各自的状态和落宫环境：

- **强强互溶**：双方均健康（入庙/入旺/无燃烧逆行）→ 相得益彰，能量通道双向畅通
- **强弱互溶**：强星接纳弱星 → 弱星受益（借强星之力缓解自身凶性），但强星可能因"要对盟友负责"而承担额外压力。如果强星自身足够稳健（Shadbala > 1.2，落角宫），则压力可控
- **弱弱互溶**：双方均受损（落陷/燃烧/逆行）→ 能量通道放大了双方的优势和弱点。这不是纯粹的坏格局，但需要后天觉察才能"化凶为吉"——当事人在相关领域（互溶涉及的宫位主题）反复碰壁后，会逐渐学会把持和转化

**落宫对互溶效果的影响**：

互溶双方的落宫位置决定了这股能量是"可控"还是"失控"：
- 双方均落角宫（1/4/7/10）→ 能量在世俗层面显现，可被感知和驾驭，即使弱弱互溶也能通过后天磨炼逐渐掌控
- 任一方落暗宫（8/12）→ 互溶的能量通道有一部分在"看不见的地方"运作，较难自我觉察，弱弱互溶在此配置下更容易表现为反复的、难以解释的困境
- 果宫（3/6/9/11）居中——可觉察但需要主动追踪

**互溶星之间的相位**：

互溶双方如果同时存在相位，相位的软硬影响通道的稳定性：
- 三合/六合（软相位）→ 能量流动顺畅，有助于缓解不良星体间的摩擦
- 刑/冲（硬相位）→ 能量流动剧烈，可能放大冲突——但同时也意味着突破力更强，"滔天大祸或万贯家财"

**接纳（Reception）补充说明**：

如果两颗星没有完全互溶、但一方落在另一方的守护星座且两者形成相位，这构成单向接纳。接纳效果弱于互溶，但仍可纳入格局扫描参考——弱星被强星接纳时获得支撑，但效果仅为互溶的约一半。

**注意**：互溶不要求双方有相位（即使不合意也能运作——经典案例已证实不合意互溶在事业/财富领域仍能释放强大能量）；但接纳必须要求相位，无相位的单向落座不构成接纳。

**接纳中的"不领情"效应**：

当 A 星接纳 B 星，但 B 星在 A 星的守护星座中落陷或处于弱势时，B 星可能"不领情"——A 提供的帮助方式不被 B 认可或接受。在分析关系宫位（7 宫主/伴侣 karaka）的接纳时，这一效应尤其重要：接纳方可能付出了很多，但被接纳方未必感激，甚至可能感到被冒犯。判断时需同时检查被接纳星在接纳星守护星座中的尊贵状态。

**特殊互溶变种（必查）**：

除正常互溶外，有两种"不等价"互溶变种必须在扫描中检查：

- **欺骗互溶（Deceptive Exchange）**：A 星落在 B 星的守护/擢升星座，但 B 星落在 A 星的**落陷或失势**星座。A 向 B 提供资源和帮助，B 非但不回报，反而将 A 置于尴尬或不利境地。这是一段不等价的能量交易——A 是"付出方"，B 是"消耗方"。在分析关系宫位（7 宫主、DK）的互溶时要特别警觉此模式。
- **拒绝互溶（Rejective Exchange）**：两颗星各自落在对方的**落陷或失势**星座。双方互相厌恶，能量通道虽然存在但"谁也不想要对方的东西"。事务容易在此模式下被取消、关系容易无疾而终。此模式比完全没有互溶更消耗——因为它制造了"不得不互动但互动质量极差"的困境。

**定位星链条（补充参考）**：

在全盘行星拆盘和互溶扫描之后，可追溯 dispositor 链条寻找"最终定位星"——即所有行星通过守护关系最终汇聚到的一颗或两颗星（互溶对）。如果存在明确的最终定位星对（互溶定位），这两颗星共同主导命盘的能量走向，其宫位掌管和落宫状态是全盘结构的"总开关"。如果不存在最终定位星（能量链不闭合），则说明命盘没有单一主导力量，各领域相对独立运作，决策时容易犹豫或缺乏统一的驱动力。

**Yogakaraka 等价分析**：

检查是否有同一颗星同时掌管角宫（1/4/7/10）和三分宫（1/5/9）。如果有，这是经典 Yogakaraka。

- 如果没有经典 Yogakaraka，不要空洞地说"没有特殊格局"就跳过
- 必须分析是否存在**功能等价组合**：互溶、多星合相、相位联动等方式形成的"准 Yogakaraka"效果
- 最低输出：一段短文解释"虽然没有经典 Yogakaraka，但 X 和 Y 的组合起到了类似的作用"

**Dhana Yoga（财富组合）扫描**：

检查 2H/11H/5H/9H 主星是否存在联动（互溶、合相、相互相位）：
- 有 → 标注为 Dhana Yoga 候选，评估参与星健康度后定级
- 无 → 说明财富主题的"独立兑现"特征（不依赖组合，靠单星实力）

### 6. 用九分盘（Navamsha chart）做兑现复核

九分盘（Navamsha chart）的作用，是复核命主盘里的承诺成熟后还能不能站住，而不是另起一套独立解读。

它主要回答五件事（详见 `术语框架.md` D9 复核五维度）：

- **内核品质**：主盘里的关键承诺到了后期是否还成立——成熟后是更稳、更专精，还是代价更高
- **安全性**：成果放在什么"仓库"里是安全的还是一碰就碎的
- **房东审计**：dispositor 链条是否支持兑现——承诺的"担保人"自己够不够强
- **变现阻力**：D1→D9 的位移是否在 6/8/12 区间，导致过程需要"绕路"
- **果实投射**：真正落果时，成果会落在 D1 的哪个宫位代表的现实领域

对每颗关键行星，复核必须显性覆盖以上五个维度中的至少四个。不能只写一段笼统的叙述就跳过。如果某维度不适用（如 Rahu/Ketu 无 dispositor 链条），要显性说明理由。

建议输出格式：

```text
——— [行星或主题] 九分盘兑现复核 ———
【现实判断】
先说成熟后会更稳、更贵、更耗，还是更能兑现。

【盘面抓手】
说明九分盘落点、宫主关系、会照和承载环境。

【使用提醒】
说明这个结论该怎么用，以及置信度受什么影响。
```

### 7.5 AK 灵魂课题（必答）

Atmakaraka（AK）是九大行星中绝对经度最高的一颗（Rahu 除外），代表灵魂这一世的"主修课"。这是 Parashari 体系的核心概念之一，不是可选加分项。

**AK 判定**：
```
遍历九大行星的绝对经度（排除 Rahu），取最大值。
如 Rahu 绝对经度最高 → 取次高行星为 AK。
```

**AK 解读至少覆盖以下三项**：

1. **AK 的身份与职责**：AK 是哪颗星 + 它在主盘掌管哪两个宫位 + 它在主盘和 D9 的落宫。
   例："AK 是金星，掌管你的 4 宫（家庭/根基）和 9 宫（信仰/远方），落在 8 宫——你的灵魂课题是'在失去和转化中找到真正的安全感'。"

2. **AK 的星宿主（Nakshatra）**：AK 所在的 Nakshatra 如何修饰灵魂课题的"学习方式"——是激烈顿悟式、渐进积累式，还是反复试错式。

3. **灵魂主线叙述**（3-5 段）：
   - AK 的本质是什么（"你这一生来学什么"）
   - 为什么这个课题对你来说这么难（宫位与行星状态给的解释）
   - 这个课题在你的关键 dasha 时期被怎样激活
   - D9 中 AK 的状态——成熟后这个课题是能毕业，还是会以另一种形式继续

**注意**：AK 解读不是"算命"，而是提供一面镜子让用户理解自己反复遇到的核心模式。用叙述语言，不用标签语言。

### 8. 可选高级模块：窗口与场域专题

当用户进一步问：

- 哪个城市更合适
- 哪个时间窗更值得推进
- 同一主题在不同地点会不会出现差异
- 某个合作、迁移、结婚或跳槽窗口该怎么缩小
