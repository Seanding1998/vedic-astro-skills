# 窗口与场域专题流程

如果需要共享术语、读盘抓手或冲突裁定口径，读取 `术语框架.md`。

本专题遵循 SKILL.md「分层推断规则」（层 0/1/2/3）。时间窗口与场域判断作为层 2 联合推断的输入参数——场域差异缩小层 0 信号的可能兑现方向。

## 这是什么

窗口与场域专题不是另外发明一张天文分盘，也不是脱离原命盘独立判断，更不是西方占星中的合盘（synastry/composite chart）。

它是一种高级专题，用来把下面三件事叠加起来看：

- 时间窗口
- 场域或地点差异
- 某个具体事件主题

也就是说，它回答的是：同一个人，在什么时间、什么地点、围绕什么主题，更容易把事情推成，或者更容易遇到摩擦。

## 适用场景

当用户明确问的是下面这些问题时，使用这个流程：

- 哪个城市更适合发展
- 什么时候搬迁、跳槽、合作、结婚更顺
- 同一主题在不同地点会不会出现明显差异
- 某个大窗口里，哪一段时间更值得主动推进
- 迁移、婚姻、事业与事件落地如何叠加判断

如果用户只是要看整张盘或单一事业、婚姻主题，不要先上窗口与场域专题。先用 `总盘.md`、`事业.md`、`婚姻.md` 建底，再决定是否升级。

## 需要的输入

优先接受其中一种：

- 当前对话里已有可用的总盘摘要
- Jagannatha Hora PDF
- JHora 导出的 markdown 报告
- 星盘截图
- 文本命盘数据

额外需要补充的输入：

- 当前或目标城市、地区、国家
- 想看的主题：事业、婚姻、迁移、合作、具体事件
- 想看的时间范围
- 如果有多个地点，最好把备选地点一次列出

如果没有可靠出生时间，要先明确告知：地点和窄时窗判断的可信度会下降。

## 分析基底

窗口与场域专题至少要站在下面这些东西上：

- 命主盘（Rashi chart）的主结构
- 九分盘（Navamsha chart）的成熟度与兑现质量
- 当前与未来相关的大运、小运
- transit / gochara
- 目标主题对应的宫位与 karaka
- 必要时的居住地、事件地、合作地差异

如果总盘还没建好，先补最低限度的底稿。不要跳过底层结构，直接给地点建议。

## 分析步骤

### 步骤一：时间扫描

1. 从 Dasha 时间线提取与目标主题高度相关的窗口段
2. 对窗口内的大运/小运做功能解读——大运管"主题"，小运管"落地"
3. 标注时间置信度——大运区间天然可信，30天内窗口需标注
4. 跨窗口时需要区分"窗口在重叠"和"窗口在分开"的情况

### 步骤二：场域比对

场域不是命主盘自带的信息，而是用户提供的外部参数。需要将场域视为"对某一条已有命盘线路的放大或抑制条件"。

1. 地理方位与宫位关系：目标地点相对于出生地的方位，与命盘中对应宫位的关联
2. 迁移盘的参考价值：以目标地点重排的盘可以作为辅助，但权重低于命主盘
3. 场域选择分类：
   - "事业型迁移"：看 10H、L10、AmK、D10
   - "学习型迁移"：看 9H、L9、Mercury、Jupiter
   - "关系型迁移"：看 7H、L7、DK、Venus
   - "疗愈型迁移"：看 12H、Moon、Saturn

### 步骤三：时间×场域联合判断

1. 对每个"备选地点+备选时间窗口"组合做权重排序
2. 每个组合给出：
   - 顺风因素（地点和时间的加持点）
   - 逆风因素（地点和时间的摩擦点）
   - 取舍逻辑（优先保什么、可以放什么）
3. 最终输出一个优先级排序 + 每个选项的取舍建议

## 输出格式

每个地点选项按以下结构输出：

```
### [地点名称]

**总体评估**：[一句话判断]

**时间窗口**：[YYYY.MM–YYYY.MM] — [触发因素]

**顺风**：
- [因素 1]
- [因素 2]

**逆风**：
- [因素 1]
- [因素 2]

**取舍建议**：[具体建议]
```

## 注意事项

- 如果出生时间精度不够（±15分钟以上），地点判断的宫位级精度会受影响——必须标注
- 不要因为没有"目标地点排盘数据"就跳过分析——用方位和宫位逻辑替代
- 不要在用户只问"适合去哪个城市"时附带全量命盘分析
- 事件判断落实到"结构层"和"窗口层"即可，不要硬推到"专名层"
